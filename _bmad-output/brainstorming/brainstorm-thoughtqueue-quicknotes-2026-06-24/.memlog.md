- (technique) started Cursed Genie
- (idea) Right-click context-menu action + key command to create a note from selected text
- (idea) Quick-capture option in the menu-bar/taskbar dropdown menu
- (idea) Auto-generate a title for each note (no manual typing)
- (idea) Auto-organize notes into categories
- (idea) Use a small local LLM to power auto-titling and auto-categorization (private, offline)
- (idea) Transient toast popup under menu-bar icon showing auto title+category with an Edit button; auto-dismisses after a user-configurable timeout
- (idea) Each .md note carries YAML frontmatter (title, category, date); filename = slug + timestamp so files never collide and are self-contained/portable
- (technique) started Ideation Relay Race
- (direction) REFRAME: app is a thin shell/interface OVER the filesystem, not a note silo. Notes are plain .md files in a local folder/file structure, fully editable in any text editor outside the app. The filesystem is the source of truth; the app creates/updates/organizes.
- (insight) Tension: this challenges the earlier YAML-frontmatter idea. Metadata may need to live in the folder/file structure itself (category=folder, title=filename) rather than inside/around files, to keep notes clean for external editing.
- (idea) Date metadata comes from the filesystem itself (created/modified timestamps) — no DB needed for dates; reinforces app-as-shell model
- (idea) App watches the store folder and live-reflects external changes (edits, moves, deletes by other tools) — true two-way sync with the filesystem
- (idea) Open-with destinations beyond Claude: Gemini (and other AI tools), Zed IDE
- (idea) Open-with can inject an '@'-style reference with the note's full file path into the target app (e.g. open Claude pre-loaded with '@/path/note.md' so the session can read the file) — not just launch, but hand off context
- (idea) Open-with comes in TWO action types: (1) Command type — run a shell command with the path as input, e.g. 'zed {path}'; (2) App+input type — open/activate an app, then inject input via simulation, either '@{path}' reference or the full pasted document text (reuses existing CGEvent keyboard-sim approach)
- (idea) Fast retrieval = fuzzy search across all note titles
- (idea) A user setting controls default click behavior on a note: (a) run the default open command, (b) view/render markdown in-app, or (c) edit raw text in-app
- (technique) started invented technique #1: The Defector — name a notes app you use/abandoned and the exact moment it failed you; that failure becomes a feature spec for yours
- (insight) Positioning gap from defection: every notes app is either 'too much' (Notion) or 'not flexible enough'. Apple Notes failure = you don't own your data. Sweet spot for this app = minimal + flexible + you fully own the data (plain local files)
- (insight) GUARDRAIL/north-star: the line isn't a single feature, it's purpose. It must stay a quick PERSONAL note app and never drift into a knowledge-base / document-management / corp tool. Scope creep = purpose creep.
- (decision) Product rule / line in the sand: say NO to any feature that requires an account or costs money. Stays free, account-less, local-only — no SaaS dependency.
- (technique) started invented technique #2: Outlive the App — imagine the app is gone/unmaintained in 5 years; what must be true of the folder so the notes stay fully usable?
- (insight) Survival test result: the only thing lost if the app dies is CONVENIENCE (quick taskbar access), never content or organization. Validates the model — app = access layer; folders/filenames/md files = durable truth. Acceptable to lose the UX, never the data.
- (decision) Radical data minimalism: the NOTES are the only thing worth persisting. Everything else (configs, search index, LLM, auto-smarts) is disposable helper functionality. The store folder should contain only notes — no app-specific clutter.
- (technique) started Job to Be Done (facilitator-chosen)
- (idea) JTBD: 'In a meeting, capture todo-list notes for later reference, so I can look things up and get people answers.' Underlying job = grab actionable todos fast mid-meeting, then retrieve to follow up. Implies markdown checkboxes/todo items + easy retrieval of action items.
- (idea) JTBD: 'When I see text I want to reference later or build into a spec/plan/document, capture it, so I can later find it and assemble it into a document.' Underlying job = collect fragments over time and accrete them into a larger doc.
- (idea) Capture supports BOTH modes: create a new standalone note, OR append the captured text onto an existing note you're already building. Serves the accretive 'grow a doc from fragments' job without bespoke doc-management features.
- (idea) Append-target selection: a popup picker to choose which note to append to
- (idea) User can quickly designate a 'working' document — a current catch-all where all captures land by default unless a different target is specified. Powerful default that serves the accretive job with near-zero friction.
- (insight) SYNTHESIS / connections: (1) The 'working document' default sink resolves the original Cursed-Genie tension — it delivers 'capture without thinking' (zero decisions) AND avoids the nameless-landfill problem, because captures accrete into a known target instead of spawning orphan notes. (2) Three ideas are really one: app-as-shell, filesystem-as-database (folder/filename/timestamp), and 'notes are the only durable thing' are a single coherent architecture — radical portability. (3) The breakthrough feature is open-with-as-context-handoff (the @path reference into Claude), not the basic launcher — it's what makes this MORE than a notes app while staying minimal. (4) Killing frontmatter to keep files clean for external editing is the move that makes the whole filesystem-shell philosophy internally consistent.
